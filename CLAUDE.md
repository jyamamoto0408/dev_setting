# 開発ガードレール — ADR付きチケット駆動ATDD

このリポジトリでは **1チケット＝1機能** を単位に、受入テスト先行（ATDD）で開発し、
重要な設計判断はADRとして記録する。文書は「背骨（常に作る）＋外部IF仕様書（常設）＋
詳細設計書（必要時のみ）」の構成で維持する。AI（Claude）は以下のルールに**必ず**従うこと。

## 絶対ルール（違反禁止）

1. **チケットなしで実装しない。** 対応するチケット（`docs/tickets/TICKET-*.md`）が
   存在し、ステータスが `approved` 以降でない限り、プロダクションコードを書かない。
   チケットがない依頼を受けたら、まずチケット作成を提案する。
   （唯一の例外: 下記「軽微修正（tweak）」の条件をすべて満たす修正）
2. **受入テストより先に実装しない。** 受入テストが存在し、かつ「失敗する」ことを
   確認してから実装を開始する（Red → Green）。
3. **承認済みの受入テストを勝手に変更しない。** テストが間違っていると判断した場合は、
   実装を止めて理由を報告し、人間の承認を得てからチケットとテストを更新する。
   テストを通すためにアサーションを弱める・skipする・削除することは禁止。
4. **チケットの「対象外」に触れない。** スコープ外のファイル・機能に変更が必要だと
   気づいたら、変更せずに報告する（別チケットの候補として提案する）。
5. **完了条件はテストのパスのみ。** 「たぶん動く」「目視で確認した」は完了ではない。
   受入テストと既存テストが全てパスして初めて完了。
6. **重要な設計判断はADRに残す。** 下記「ADRトリガー」に該当する判断をしたら、
   実装完了前に `docs/adr/` にADRを起票する。
7. **実装は計画書（PLAN）から始める。** 新規・修正を問わず、チケット群の実装に入る前に
   `docs/plans/PLAN-NNN`（実装計画書: アーキテクチャ・開発順序・作成規則・チケット分割）を
   作成し、承認を得る。プランモードで立てた計画も破棄せず PLAN として保存する。
   計画書を飛ばして直接コードを直さない。
8. **外部IF（API・DBスキーマ）の変更は特別扱い。** `docs/interfaces/` の仕様書を
   **実装前に**更新し、ADRを起票する。後方互換性を壊す変更（エンドポイント削除、
   レスポンス構造変更、カラム削除・型変更など）は、その旨を明示して人間の承認を得る。
9. **開発過程でできた文書はすべて残す。** 要件定義書・計画書・設計書・レビュー記録は
   成果物の一部。削除や上書きで検討の経緯を消さない。変更は変更履歴への追記で行う。
10. **レビューを経ずにチケットをクローズしない。** 実装完了後、レビュー（`docs/reviews/`）で
    must-fix 指摘がゼロになるまで `done` にしない。指摘対応の修正でも絶対ルール1〜8は
    そのまま適用される（テストの改ざん禁止・スコープ制御・外部IF規律）。
    スコープ外の指摘を勝手に直さない（別チケット候補として記録する）。

## ドキュメント体系

### 背骨（常に作る）

```
要件定義書 (docs/requirements/requirements.md)
  ← 目的・要件を最初に文書化し、壁打ちでブラッシュアップする「生きた文書」。
    世代別ファイルを増やさず、1つの文書＋変更履歴で最新を維持する。
  ↓
計画書 (docs/plans/PLAN-NNN)
  ← 実装計画: アーキテクチャ・開発順序・作成規則・チケット分割。
    チケット→ATDD→実装→ADRの一連をどう回すかを定める。
    プランモードで立てた計画はここに保存する。
  ↓
チケット (docs/tickets/TICKET-NNN)
  ← 計画の開発順序に沿って、チケットごとにATDDの6フェーズを回す
  ↓
ADR (docs/adr/ADR-NNNN)
  ← 実装中の重要な設計判断を記録
  ↓
レビュー記録 (docs/reviews/REVIEW-NNN)
  ← 実装のレビュー指摘・対応・再レビューの経緯を記録（1チケット1文書）
```

開発中に修正・仕様変更が出たら: 計画書（type: fix）を作成 → 影響する文書を更新 →
チケット化、の順で進める。コードを直接直さない。

### 常設の外部IF仕様書（docs/interfaces/）

外部に公開する・他コンポーネントと共有する境界は「生きた仕様書」として常設維持する。

- `API.md` — APIリファレンス（エンドポイント・リクエスト/レスポンス・エラー）
- `DB.md` — DBスキーマ（テーブル・カラム・リレーション・制約）

**変更は計画書＋ADR経由のみ。** 実装がこれらの文書とずれた状態を放置しない
（チケット完了チェックで検証する）。サービスとして外部提供する場合、これらは
「勝手に変えられない契約」であることを常に意識する。

### 任意層: 詳細設計書（docs/designs/DESIGN-NN）

毎回は書かない。以下のトリガーに該当するときだけ書く。

- 要件定義と計画書の間の飛躍が大きく、実装者（AI）が迷う複雑な機能
- 複数の計画書にまたがって参照される共通基盤・アルゴリズム
- 状態遷移・データフローが複雑で、文章より図表で固めるべきもの

### 文書の規律

1. **理由を書く。** すべての文書に「なぜこの設計・決定に至ったか」を書く。結論だけの
   文書は不完全。検討した代替案と採らなかった理由も残す。
2. **変更履歴を残す。** 文書を変更したら変更履歴に日付・変更内容・理由を追記する。
   壁打ちでの方針転換も記録対象（何をどう変えたか、なぜ変えたか）。
3. **文書ファースト。** 実装と文書がずれたら文書を先に更新する。上流文書と矛盾する
   実装依頼を受けたら、指摘して文書の更新を提案する。
4. **トレーサビリティ。** 要件ID（REQ-F-NN / REQ-N-NN）→ 計画書 → チケット → ADR が
   たどれるように、各文書は上流のIDを明記する。

## ワークフロー（6フェーズ）

各チケットは承認済みの計画書（PLAN-NNN）に紐づき、以下の順で進める。フェーズを飛ばさない。

| フェーズ | やること | 完了条件（次に進む条件） |
|---|---|---|
| 1. 起票 | `/ticket-new` でチケット作成 | 人間が内容を確認し `status: approved` にする |
| 2. 受入テスト | `/ticket-start` で受入テストを作成 | テストが**失敗する**ことを確認し、人間がテスト内容を承認 |
| 3. 実装 | チケットのスコープ内のみ実装 | 受入テスト＋既存テスト全パス |
| 4. ADR | 該当する設計判断をADR化 | ADRトリガーに該当なしと明記 or ADR起票済み |
| 5. レビュー | `/ticket-review` でレビューし指摘対応 | must-fix 指摘ゼロ、人間が対応方針を承認 |
| 6. 完了 | `/ticket-done` でクローズ | チケットの完了条件チェックリスト全て✅、`status: done` |

フェーズ1・2の末尾、およびフェーズ5の指摘対応方針の提示時には
**必ず人間の承認待ちで停止する**。承認を推測して先に進まない。

## レビューの規律

1. **実装者とレビュアーを分ける。** レビューは実装を行ったセッションとは
   **別の新しいセッション（またはサブエージェント）**で実施する。実装の記憶・思い込みを
   持たない状態で、チケット・受入テスト・diffだけを入力にレビューさせる。
2. **共通レビュー観点（言語・スタック非依存）。** 最低限以下を確認する。
   - 完了条件の充足: 各ACが実装とテストで本当に満たされているか
   - テストの品質: アサーションが弱くないか、テストが実装の写しになっていないか、
     境界値・異常系があるか
   - スコープ: チケット範囲外の変更が混ざっていないか
   - 外部IF: `docs/interfaces/` と実装が一致しているか
   - セキュリティ: 入力検証、認証・認可、秘密情報のハードコード、インジェクション
   - ガードレール違反: 絶対ルール・計画書の「作成の規則」からの逸脱
3. **プロジェクト固有の観点は計画書に書く。** 技術スタック固有のチェック項目は
   PLAN の「作成の規則」に定義し、レビュー時にそれを読み込んで観点に加える。
4. **指摘は3分類で管理する。**
   - `must` — 修正するまでクローズ不可（バグ、AC未達、セキュリティ、ガードレール違反）
   - `should` — 対応推奨。今回対応しない場合は理由を記録して人間が判断
   - `nice` — 任意。対応しないことをデフォルトとし、別チケット候補として記録可
5. **指摘対応も本編と同じルールで行う。** must の修正はスコープ内で行い、テスト全パスを
   維持する。受入テストの変更が必要な指摘は絶対ルール3の承認手続きを踏む。
   スコープ外に及ぶ指摘は直さずに別チケット候補として記録する。
6. **修正後は再レビューする。** must 指摘が全て解消するまで繰り返す。再レビューの結果も
   同じレビュー記録に追記する（ラウンド1、ラウンド2、…）。
7. **指摘ゼロでも記録を残す。** 「何を確認して問題なしと判断したか」も経緯の一部。
8. **繰り返される指摘はルールに昇格させる。** 同種の指摘が複数チケットで出たら、
   計画書の「作成の規則」または本文書への追記を提案する（レビューからの継続的改善）。

## ADRトリガー（1つでも該当したらADRを書く）

- 新しいライブラリ・フレームワーク・外部サービスを導入した
- データモデル / DBスキーマ / APIの公開インターフェースを決めた・変えた
- 複数の実装方式を比較して選択した（採らなかった案があるとき）
- 性能・セキュリティ・互換性のためにトレードオフを受け入れた
- 既存の設計・慣習から意図的に逸脱した

逆に、変数名・関数分割・軽微なリファクタ等の局所判断はADR不要。

## スコープ制御

- 1チケットの変更は目安として **±400行以内**。超えそうなら分割を提案する。
- チケットに列挙されたファイル/モジュール以外を変更する場合は、変更前に理由を報告する。
- リファクタリングは実装と同一チケットで行わない（別チケットにする）。

## 軽微修正（tweak）— 唯一の軽量レーン

文言変更・レイアウト調整など、計画書・チケットを作るほどでない修正のための例外フロー。
`/tweak` で実施し、記録は `docs/TWEAKS.md` に1行追記する（1件1ファイルにしない）。

### 使える条件（**すべて**満たすこと）

- 振る舞い・ロジックの変更がない（文言、レイアウト、スタイル、コメント等の変更）
- 外部IF（API・DBスキーマ）に触れない
- 受入テスト・既存テストの修正が不要（既存テストは全パス維持）
- 変更が小さい（目安: 数ファイル・数十行以内）
- **迷ったら軽微ではない方に倒し、人間に確認する**
  （例: バリデーションメッセージの文言は仕様の一部の可能性がある）

### 運用ルール

- 1つでも条件から外れたら通常フロー（`/plan-new` type: fix）へ。勝手に tweak で進めない。
- **要件定義書の変更を要する修正はユーザーが明言する。** AI側でも要件・仕様に影響すると
  判断したら tweak にせず、その旨を指摘して通常フローを提案する。
- tweak でも既存テストの全パスを確認してから記録・報告する。
- 同じ箇所への tweak が繰り返されたら、軽微でない兆候としてチケット化を提案する。

## テストの規律

- 受入テストはチケットの「完了条件」と1対1で対応させる（各条件に最低1テスト）。
- テストは外部環境（ネットワーク、時刻、乱数）に依存させない。依存する場合はモック/固定化。
- 既存テストが落ちたら、まず自分の変更を疑う。既存テストの修正が必要な場合は理由を報告して承認を得る。

## ファイル配置

```
docs/
├── TWEAKS.md             # 軽微修正ログ（追記のみ。tweakレーンの記録先）
├── requirements/
│   ├── _TEMPLATE.md      # 要件定義書テンプレート
│   ├── requirements.md   # プロジェクトの要件定義書（生きた文書、履歴を残す）
│   └── archive/          # 節目ごとのスナップショット（編集禁止）
│       └── requirements-01.md
├── plans/
│   ├── _TEMPLATE.md      # 実装計画書テンプレート（新規・修正共通）
│   └── PLAN-NNN-GG-slug.md
├── interfaces/           # 常設の外部IF仕様書（変更は計画書＋ADR経由のみ）
│   ├── _TEMPLATE-API.md
│   ├── _TEMPLATE-DB.md
│   ├── API.md
│   ├── DB.md
│   └── archive/          # 節目ごとのスナップショット（編集禁止）
├── designs/              # 任意層（トリガー該当時のみ）
│   ├── _TEMPLATE.md      # 詳細設計書テンプレート
│   └── DESIGN-NN-GG-slug.md
├── tickets/
│   ├── _TEMPLATE.md      # チケットテンプレート
│   └── TICKET-NNN-GG-slug.md
├── reviews/
│   ├── _TEMPLATE.md      # レビュー記録テンプレート
│   └── REVIEW-NNN-GG-slug.md   # 番号は対象チケットに合わせる（TICKET-003 → REVIEW-003）
└── adr/
    ├── _TEMPLATE.md      # ADRテンプレート
    └── ADR-NNNN-GG-slug.md
```

- チケット番号は連番（TICKET-001, 002, ...）。ADR・PLANも連番。
  REVIEWは対象チケットと同番号。連番は世代をまたいでもリセットしない。
- ADRは追記のみ。決定を覆すときは古いADRを編集せず、新しいADRで `superseded by` を記録する。

## 世代管理（いつの開発の文書かをファイル名で識別する）

- **開発世代（GG）**: 最初の開発を `01` とし、プロジェクト完成後のアップデート開始などの
  節目で +1 する。現行世代は要件定義書の frontmatter `generation:` で管理する。
- **連番文書のファイル名**: `種別-連番-世代-slug.md`（例: `TICKET-013-02-user-auth.md`
  ＝ 世代02のアップデート開発で作られたチケット013）。
- **節目の手順**（アップデート開発を始める前に必ず行う）:
  1. 生きた文書のスナップショットを `archive/` に現世代番号付きで保存する
     （例: `docs/requirements/archive/requirements-01.md`、`docs/interfaces/archive/API-01.md`）。
     スナップショットは凍結文書であり、以後編集禁止。
  2. 要件定義書の `generation:` をインクリメントし、変更履歴に節目（何が完成し、
     何のアップデートを始めるか）を記録する。
  3. 以降に作成する連番文書には新しい世代番号を付ける。既存文書はリネームしない。

## 報告の作法

- 各フェーズ完了時に、何をしたか・テスト結果・次に人間の判断が必要な点を簡潔に報告する。
- テストが通らない状態で「完了」と報告しない。ブロックされたら正直にブロック内容を報告する。
