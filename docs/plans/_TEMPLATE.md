# PLAN-NNN: タイトル

チケット・受入テスト（ATDD）・実装・ADRの一連をどう進めるかを定めた**実装計画書**。
新規開発・修正のどちらでも、実装に入る前に作成し承認を得る。
プランモード（plan.md）で立てた計画は破棄せず、この形式で保存する。

## 目的・背景

- feature: 要件定義書のどの要件（REQ-F-NN）を実装するか。
- fix: 何が起きて、なぜ修正が必要か。放置した場合の問題。

## 実装アプローチ・アーキテクチャ

どういう構成・方式で実現するか。
**なぜこのアプローチにしたか、検討した代替案と採らなかった理由も必ず書く。**
ADRトリガーに該当する判断は ADR に起票してここからID参照する。
複雑さが詳細設計書のトリガー（CLAUDE.md参照）に該当する場合は DESIGN-NN を起こし、
`design:` に記載する。

## 外部IF（API・DB）への影響

`docs/interfaces/API.md` / `DB.md` に影響するか**必ず判定する**。影響なしでもその旨を明記。

- API: なし / あり（変更内容。**実装前に** API.md を更新し、ADRを起票する）
- DB: なし / あり（変更内容。**実装前に** DB.md を更新し、ADRを起票する）
- 後方互換性を壊す変更: なし / あり（→ 人間の明示承認が必要）
