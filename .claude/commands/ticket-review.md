対象チケット: $ARGUMENTS

実装のレビューと指摘対応を以下の手順で進めよ。

## 前提確認

1. チケットが `in-progress` で、受入テスト＋既存テストが全パスしていることを確認する
   （パスしていなければレビューに入らず、フェーズ3が未完了である旨を報告して停止）。

## レビュー実施（実装者と分離する）

2. **このセッションで実装を行った場合は、自分でレビューしない。** サブエージェント
   （Agentツール）に以下だけを入力として渡し、レビューさせる:
   チケット本文・受入テスト・変更diff・計画書の「作成の規則」・`docs/interfaces/`。
   実装時の経緯や意図は渡さない（先入観を持たせない）。
   実装がこのセッションでない場合（レビュー専用セッション）は直接レビューしてよい。
3. CLAUDE.md「レビューの規律」の共通観点＋PLANのプロジェクト固有観点でレビューし、
   `docs/reviews/_TEMPLATE.md` を元に `docs/reviews/REVIEW-NNN-<GG>-<slug>.md`
   （番号・世代・slugは対象チケットのファイル名に合わせる）を作成する。指摘は must / should / nice に分類し、根拠（ファイル:行）を必ず書く。
   既にREVIEW-NNNが存在する場合は新ラウンドとして追記する。

## 指摘対応

4. 指摘一覧と対応方針（must=修正 / should=対応するか否かの提案 / nice=記録のみ）を提示し、
   **人間の承認を待って停止する。**
5. 承認後、must と対応が承認された should を修正する。
   - 修正もチケットのスコープ内・テスト全パス維持・受入テスト改ざん禁止のルール下で行う。
   - スコープ外に及ぶ指摘は修正せず「別チケット候補」に記録する。
6. 修正後、**再レビュー**（手順2〜3と同様、新ラウンドとして追記）を行い、
   must がゼロになるまで繰り返す。デグレ（修正による新たな問題）も確認する。

## 完了

7. must ゼロになったら REVIEW の `status` を `approved` にし、指摘対応の要約・
   別チケット候補・ルール昇格提案（あれば）を報告して `/ticket-done` を提案する。
