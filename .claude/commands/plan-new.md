チケット・受入テスト（ATDD）・実装・ADRの一連をどう進めるかを、実装前に
計画書として文書化せよ。**コードはまだ書かないこと。**

対象: $ARGUMENTS

1. type を判定する: 新規開発（feature）か、開発中の修正・仕様変更（fix）か。
2. **計画前インタビュー**: 実装アプローチ・開発順序を左右する「重要な曖昧さ」
   （実装方向、検証方法、データの扱い、権限・セキュリティ、リリース手順など）が
   あれば、計画を書き切る前に CLAUDE.md「質問の規律」に従って質問し解消する
   （実現可能な選択肢を最大3つまで、推奨案とトレードオフ付きで提示）。
   調べれば分かることは質問せず調査し、重要な曖昧さがなければ質問しない。
3. `docs/plans/` を確認して次のPLAN番号を決め、`docs/plans/_TEMPLATE.md` を元に
   `docs/plans/PLAN-NNN-<GG>-<slug>.md` を作成する
   （GG＝要件定義書の frontmatter `generation:` の現行世代番号）。
   - **プランモードで計画を立てていた場合は、その内容を破棄せずこのPLANに反映して保存する。**
4. feature の場合: `docs/requirements/requirements.md` が approved で、対象の要件ID
   （REQ-F-NN）が定義済みであることを確認し（なければ停止して報告）、以下を埋める。
   - **実装アプローチ・アーキテクチャ**（なぜそのアプローチか、代替案と採らなかった理由）
   - **開発順序**（ステップ⇔チケットの対応。依存関係と順序の理由）
   - **作成の規則**（テスト戦略・命名・使用ライブラリ等、この実装で守る規約）
   - 複雑さが詳細設計書のトリガーに該当する場合は `/design-new` を先に提案する。
5. fix の場合: 上記に加えて「きっかけ」「なぜ必要か」を書き、
   **影響を受ける上流ドキュメント**（要件定義書・interfaces・DESIGN・既存チケット）を
   特定する。影響なしの場合もその旨を明記する。
6. **外部IF（API・DB）への影響を必ず判定する。** `docs/interfaces/API.md` / `DB.md` に
   影響がある場合はその変更内容を計画書に書き、**実装前に仕様書を更新し ADR を起票する**
   段取りを開発順序に組み込む。後方互換性を壊す変更はその旨を明示する。
7. 計画書を提示し、承認（status: approved）を求めて停止する。
8. 承認後: （fixなら先に上流ドキュメントを更新し、変更履歴に PLAN-NNN を理由として記載）
   開発順序に沿って `/ticket-new` でチケットを起票し、PLANの `tickets:` と
   各チケットの関連ドキュメント欄を相互リンクする。
9. 対象チケットが全て done になったら PLAN の `status` を `done` にする。
