# ADR付きチケット駆動ATDD — 開発セット（2層＋外部IF常設）

Claude Code（Opus / Sonnet）で「要件定義 → 計画 → 1チケット＝1機能のATDD」を
文書駆動で行うためのガードレール一式。Codex CLI向けは `codex/` フォルダにある。

**このフォルダはガードレールのマスターリポジトリ。** ここでは開発しない。
各プロジェクトは独立フォルダ（各自gitリポジトリ）で作り、`sync-guardrails.ps1` で
ガードレールを配布する（下記「複数プロジェクトの運用」参照）。

## 複数プロジェクトの運用（忘れたらここを読む）

### 構成の原則

- **1プロジェクト＝1独立フォルダ＝1gitリポジトリ**。dev_setting の配下には作らない
  （AIが隣のプロジェクトに触れないよう物理的に分離する。チャット履歴も自然に分かれる）。
- dev_setting は**マスター**。ガードレールの修正は必ずここで行い、スクリプトで配布する。
  各プロジェクト側で CLAUDE.md やコマンドを直接編集しない（次回の同期で消える）。

### 新規プロジェクトを始めるとき

```powershell
cd C:\Users\Fantas03\Documents\dev_setting
.\sync-guardrails.ps1 -Init C:\path\to\new-project          # 両方（既定）
.\sync-guardrails.ps1 -Init C:\path\to\new-project -Type claude   # Claude Codeのみ
```

docs構成の作成＋ガードレール配布＋ `projects.txt` への登録まで行われる。
その後、プロジェクト側で `git init` → Claude Code / Codex を**プロジェクトフォルダで起動**
→ `/req-new` から開発を始める。

### ガードレールを更新したとき

```powershell
cd C:\Users\Fantas03\Documents\dev_setting
.\sync-guardrails.ps1           # projects.txt の全プロジェクトへ一括配布
.\sync-guardrails.ps1 -DryRun   # 配布内容の事前確認
```

配布されるのは**ルール・コマンド/スキル・テンプレート・使い方ガイドのみ**。各プロジェクトの
実文書（requirements.md・チケット・ADR・TWEAKS.md 等）には触れない。
この README.md は各プロジェクトには `GUARDRAILS.md`（Codex版は `GUARDRAILS-codex.md`）と
いう名前で配布される。プロジェクト自身の README.md はプロジェクト側で自由に作ってよい。

### 改善の還流（プロジェクト → マスター）

レビューで「繰り返される指摘はルールに昇格させる」提案が出たら:
プロジェクト側で直接ルールを直さず、**dev_setting のマスターに反映 → 一括配布**の順で行う。
これで1プロジェクトでの学びが全プロジェクトに広がる。

## 文書構成の考え方

個人＋AI開発で文書メンテが破綻しないよう、文書は3種類に分ける。

| 層 | 文書 | 作るタイミング |
|---|---|---|
| **背骨（常に作る）** | 要件定義書 → 計画書 → チケット → ADR・レビュー記録 | すべての開発で |
| **外部IF仕様書（常設）** | `docs/interfaces/API.md` / `DB.md` | 生きた文書として常時維持。変更は計画書＋ADR経由のみ |
| **任意層** | 詳細設計書（DESIGN-NN） | 複雑な機能・共通基盤などトリガー該当時のみ |

外部にサービス提供する場合、API・DBスキーマは「勝手に変えられない契約」になるため、
この2つだけは例外的に厳格管理する（実装前更新・ADR必須・互換性破壊は明示承認）。

※ 旧「仕様書・基本設計書（SPEC）」層は2026-07-07に廃止（計画書・interfaces に役割を移管）。

## 構成

| ファイル | 役割 |

| `docs/interfaces/_TEMPLATE-API.md` / `-DB.md` | 常設の外部IF仕様書 |

| `docs/tickets/_TEMPLATE.md` | チケットテンプレート |
| `docs/reviews/_TEMPLATE.md` | レビュー記録（指摘・対応・再レビューを全ラウンド記録） |

## 全体フロー

```
/req-new <目的・要件>        要件定義書を作成 → AIと壁打ちでブラッシュアップ
      ↓ 承認                 （決定の経緯は変更履歴に残る）
/plan-new <実装範囲>         実装計画書: アーキテクチャ・開発順序・規則・チケット分割
      ↓ 承認                 （プランモードの計画はここに保存。外部IF影響も判定）
/ticket-new → /ticket-start → /ticket-done     ← チケットごとにATDDの5フェーズ
      ↓
（開発中に修正が出たら）
/plan-new <何が起きたか>     計画書(type: fix) → 上流文書を更新 → 修正チケット化

（複雑な機能だけ）
/design-new <対象>           詳細設計書（トリガー該当時のみ）

（文言・レイアウト等の軽微修正だけ）
/tweak <修正内容>            軽微判定 → 修正 → テスト全パス確認 → docs/TWEAKS.md に1行記録
                             ※振る舞い・外部IF・テストに触れる場合は通常フローへ
```

チケット単位のライフサイクル:

```
/ticket-new ユーザー登録APIを作りたい
  → チケットが draft で作成される
  → 人間がレビューし status: approved に変更     ← ゲート①

/ticket-start TICKET-001
  → 受入テストが作成され「失敗する」ことを確認
  → 人間がテスト内容を承認                        ← ゲート②
  → AIが実装し、全テストパスまで修正
  → 設計判断があればADRを起票（/adr-new）

/ticket-review TICKET-001（実装とは別セッション/サブエージェントで実施）
  → must / should / nice に分類した指摘＋対応方針を提示
  → 人間が対応方針を承認                          ← ゲート③
  → must を修正 → 再レビュー → must ゼロまで繰り返す
  → 経緯は docs/reviews/REVIEW-001 に全ラウンド記録

/ticket-done TICKET-001
  → 完了条件・レビューapproved・スコープ・外部IF一致・ADR記録をチェックしてクローズ
```

## ガードレールの要点

- **人間の承認ゲート**: 要件・計画・チケット・受入テスト・レビュー対応方針で
  AIは承認待ちで必ず停止する。
- **レビューの分離**: レビューは実装したセッションとは別のセッション/サブエージェントで
  実施する（自己レビューバイアスの回避）。指摘は must/should/nice で管理し、
  must ゼロになるまでクローズ不可。繰り返される指摘はルールへの昇格を提案する。
- **計画書ファースト**: 新規・修正を問わず、実装前に計画書（PLAN）を作る。
  プランモードで立てた計画も破棄せず保存する。
- **外部IFの厳格管理**: API/DBスキーマの変更は実装前に `docs/interfaces/` を更新し
  ADRを起票する。互換性を壊す変更は明示承認が必要。チケット完了時に一致を検証する。
- **経緯を残す**: すべての文書に「なぜこの設計に至ったか」「採らなかった代替案」を書き、
  変更は変更履歴への追記で行う。壁打ちでの方針転換も記録する。文書は削除しない。
- **トレーサビリティ**: 要件ID（REQ-F-NN）→ 計画書 → チケット → ADR がたどれる。
- **世代管理**: 連番文書は `TICKET-013-02-slug.md`（連番-世代）形式で、いつの開発かが
  ファイル名で分かる。完成→アップデートの節目は `/milestone` で、生きた文書
  （要件定義書・API/DB仕様書）のスナップショットを `archive/` に凍結保存してから
  世代番号を+1する。過去の文書はリネームせずそのまま残る。
- **テストの改ざん禁止**: 承認後の受入テストを弱める・skipする・消すことは禁止。
- **完了＝テストパス**のみ。目視確認や「たぶん動く」では done にできない。
